Управление доступом в РИМ используется для контроля того, кто может взаимодействовать с ботом и может ли входящий запрос быть передан в прикладную обработку.

Внешний пользователь приходит в РИМ не как пользователь 1С, а как пользователь конкретного мессенджера: Telegram, VK или другого канала. Поэтому РИМ хранит отдельные профили пользователей мессенджеров, связывает их с сессиями и при необходимости сопоставляет с пользователями информационной базы.

Проверка доступа выполняется после подготовки входящего запроса и до вызова прикладного обработчика бота.

## 🔎 Общая схема

В упрощённом виде проверка доступа находится внутри пайплайна обработки входящего запроса:

```text
[РИМ: очередь входящих запросов]
          ↓
[РИМ: менеджер входящих запросов]
          ↓
[РИМ: контекст выполнения]
          ↓
[РИМ: контекст запроса]
          ↓
[РИМ: сессия пользователя]
          ↓
[РИМ: аутентификация и проверка доступа]
          ↓
[Бот / прикладная логика]
```

Конкретная интеграция не принимает решение о доступе пользователя. Она помогает РИМ получить данные внешнего события и инициализировать контекст запроса.

Решение о доступе принимает внутренний механизм РИМ на основании настроек канала бота, данных сессии и профиля пользователя мессенджера.

## 👤 Внешний пользователь

Пользователь мессенджера определяется внешним идентификатором, который приходит из конкретной платформы.

Для проверки доступа РИМ использует данные внешнего пользователя, чата, бота, канала и текущей сессии.

Внешний пользователь не обязан сразу быть пользователем информационной базы. В зависимости от способа доступа он может работать анонимно, проходить аутентификацию по паролю, допускаться по заранее разрешённому профилю мессенджера или связываться с пользователем ИБ.

## 📇 Профиль пользователя мессенджера

Профиль пользователя мессенджера — это внутренняя запись РИМ, которая связывает пользователя внешнего мессенджера с системой.

Профиль нужен для того, чтобы РИМ мог повторно узнавать пользователя при следующих обращениях и не выполнять полную идентификацию каждый раз.

Если способ доступа требует идентификации, РИМ может найти существующий профиль или создать новый активный профиль после успешной аутентификации.

## 🗂️ Группы профилей

Группы профилей используются в сценарии доступа по разрешённым профилям.

Администратор может заранее определить группу пользователей мессенджера, которым разрешено взаимодействовать с ботом. При проверке доступа РИМ определяет профиль пользователя и проверяет, входит ли он в разрешённую группу.

Такой способ подходит для закрытых ботов, где доступ должен быть ограничен заранее известным списком пользователей.

## 💬 Сессия пользователя

Сессия пользователя хранит состояние взаимодействия пользователя с ботом в рамках конкретного чата.

Сессия определяется связкой:

* бот;
* идентификатор чата;
* идентификатор пользователя в мессенджере.

В сессии хранится состояние доступа и текущего диалога: прошёл ли пользователь проверку, находится ли он в процессе аутентификации и какой шаг должен быть выполнен следующим.

## 🔑 Состояние аутентификации

Для многошаговых способов доступа РИМ использует состояние сессии, чтобы продолжать диалог аутентификации между несколькими входящими запросами.

Например, при доступе по пользователю информационной базы или по одноразовому коду пользователь может сначала ввести имя пользователя, а затем пароль или временный код. Между этими шагами РИМ сохраняет в сессии, на каком этапе находится пользователь и какое действие ожидается следующим.

Благодаря этому каждый новый входящий запрос может быть обработан не как отдельное независимое сообщение, а как продолжение текущего сценария аутентификации.

Если пользователь начинает взаимодействие заново, например отправляет команду начала (`старт` или `/start`), РИМ может перезапустить сценарий аутентификации и снова начать проверку доступа с первого шага.
## 🛂 Способы доступа к боту

Способ доступа задаётся в настройках канала бота и определяет, как РИМ будет проверять пользователя перед выполнением прикладного обработчика.

### Анонимный доступ

При анонимном доступе РИМ разрешает выполнение обработчика без идентификации пользователя.

Такой режим подходит для открытых ботов, где не требуется связывать пользователя мессенджера с внутренним пользователем или заранее разрешённым профилем.

### Доступ по разрешённым профилям

При доступе по разрешённым профилям РИМ проверяет, есть ли активный профиль пользователя мессенджера в разрешённой группе профилей.

Если профиль входит в группу, доступ разрешается. Если профиль отсутствует, пользователь получает сообщение об отказе.

### Доступ по паролю

При доступе по паролю РИМ запускает диалог аутентификации и просит пользователя ввести пароль доступа к боту.

Если пароль введён корректно, РИМ находит или создаёт активный профиль пользователя мессенджера, фиксирует успешную аутентификацию в сессии и разрешает дальнейшую обработку.

Если пароль неверный, РИМ повторно запрашивает пароль.

### Доступ по пользователю ИБ

При доступе по пользователю информационной базы РИМ запрашивает имя пользователя 1С, затем пароль пользователя ИБ.

Если пользователь найден и пароль корректный, РИМ связывает профиль пользователя мессенджера с пользователем информационной базы, фиксирует успешную аутентификацию и разрешает обработку запроса.

Этот способ позволяет связать внешнего пользователя мессенджера с внутренним пользователем 1С.

### Доступ по одноразовому коду

При доступе по одноразовому коду РИМ запрашивает имя пользователя, создаёт временный код и регистрирует его для пользователя.

После ввода корректного кода РИМ создаёт или обновляет профиль пользователя мессенджера, связывает его с пользователем ИБ и фиксирует успешную аутентификацию в сессии.

Такой способ позволяет подтверждать доступ через одноразовый код без передачи постоянного пароля пользователя в мессенджере.

## 🚦 Результаты проверки доступа

После проверки доступа РИМ получает один из основных результатов:

* **разрешено** — пользователь может продолжить работу, запрос передаётся прикладному обработчику бота;
* **запрещено** — пользователь не имеет доступа, прикладной обработчик не выполняется;
* **ожидание действий пользователя** — пользователь находится в процессе аутентификации, обработчик бота пока не вызывается.

Если доступ разрешён, РИМ помещает данные пользователя в контекст запроса и выполняет обработчик бота.

Если доступ запрещён или требуется продолжить диалог аутентификации, обычная прикладная обработка запроса не выполняется.

## ⛔ Отказ в доступе

Если доступ запрещён, РИМ не передаёт запрос в обычную прикладную обработку.

В зависимости от способа доступа РИМ может:

* отправить сообщение об отказе в доступе;
* предложить пользователю пройти аутентификацию;
* продолжить текущий диалог аутентификации;
* завершить обработку входящего запроса.
