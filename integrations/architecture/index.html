    🏗️ Архитектура

    Эта страница показывает архитектурное устройство интеграционного механизма РИМ: какие компоненты участвуют в обмене сообщениями, как между ними проходит поток данных и где разделяется ответственность между РИМ, интеграцией мессенджера и прикладной логикой бота.

    Интеграционный механизм связывает мессенджеры с прикладной логикой ботов. Событие из мессенджера преобразуется во входящий запрос РИМ, проходит внутреннюю обработку, передаётся боту и может привести к постановке исходящего сообщения в очередь отправки.

    🧭 Принцип работы

    РИМ отделяет прикладную логику бота от особенностей конкретного мессенджера.

    Интеграция мессенджера работает с API внешней платформы: получает события, преобразует данные мессенджера во внутренний формат РИМ и отправляет исходящие сообщения обратно через API мессенджера.

    Внутренние механизмы РИМ отвечают за очереди, контексты обработки, сессии, аутентификацию, проверку доступа и передачу запроса в прикладную логику бота.

    Бот при этом работает не с Telegram, VK или другим API напрямую, а с внутренней моделью событий, сообщений и контекстов РИМ.

    В упрощённом виде интеграционный механизм можно представить так:

    [Пользователь]
      ↓
[Мессенджер]
      ↓
[РИМ: интеграция мессенджера]
      ↓
[РИМ: очередь входящих запросов]
      ↓
[РИМ: менеджер входящих запросов]
      ↓
[Бот / прикладная логика]
      ↓
[РИМ: очередь сообщений к отправке]
      ↓
[РИМ: обработчик очереди отправки]
      ↓
[РИМ: интеграция мессенджера]
      ↓
[Мессенджер]
      ↓
[Пользователь]

    Интеграция мессенджера находится на границе между мессенджером и внутренними механизмами РИМ. Она получает данные из API мессенджера и передаёт их в РИМ как входящий запрос.

    Дальше запрос обрабатывается внутренними механизмами РИМ: попадает в очередь входящих запросов, выбирается менеджером входящих запросов, для него определяются бот, адресат, сессия и результат проверки доступа, после чего передаётся в прикладную логику бота.

    Ответ бота также не отправляется напрямую в мессенджер. Сначала он записывается в очередь сообщений к отправке, затем обрабатывается отдельным механизмом отправки и только после этого передаётся интеграции мессенджера для отправки через API мессенджера.

    🧱 Основные компоненты

    В интеграционном механизме участвуют несколько основных компонентов и этапов обработки:

    • мессенджер — внешняя платформа, через которую пользователь взаимодействует с ботом;
    • интеграция мессенджера — реализация подключения к API конкретного мессенджера;
    • очередь входящих запросов — очередь записей с входящими запросами, полученными от мессенджеров;
    • менеджер входящих запросов — механизм, который выбирает записи из очереди входящих запросов и запускает их обработку;
    • бот / прикладная логика — сценарии, команды и бизнес-логика, которые реагируют на входящий запрос;
    • очередь сообщений к отправке — очередь записей с исходящими сообщениями, ожидающими отправки в мессенджер;
    • обработчик очереди отправки — механизм, который выбирает сообщения из очереди, выполняет отправку и фиксирует результат.

    🔁 Роль очередей в архитектуре

    Очереди в РИМ используются как промежуточный слой между получением входящих событий, выполнением прикладной логики и отправкой исходящих сообщений.

    РИМ использует два основных направления очередей:

    • очередь входящих запросов — хранит входящие данные, полученные от внешнего мессенджера;
    • очередь сообщений к отправке — хранит исходящие сообщения, которые должны быть отправлены пользователю через мессенджер.

    Очереди позволяют не выполнять весь цикл обработки прямо в момент получения события от внешнего сервиса и не отправлять ответы напрямую из прикладной логики бота.

    Благодаря этому:

    • интеграция мессенджера не смешивается с бизнес-логикой бота;
    • входящий запрос сначала фиксируется в системе, а затем обрабатывается внутренними механизмами РИМ;
    • исходящее сообщение сначала записывается в очередь отправки, а затем отправляется отдельным обработчиком;
    • ошибки обработки и отправки можно фиксировать и разбирать отдельно;
    • записи очередей можно хранить или удалять в зависимости от режима хранения.

    Очереди находятся внутри интеграционного механизма РИМ и связывают транспортный слой, внутреннюю обработку и прикладную логику бота.

[Очередь входящих запросов]
          ↓
[Менеджер входящих запросов]
          ↓
[Бот / прикладная логика]
          ↓
[Очередь сообщений к отправке]
          ↓
[Обработчик очереди отправки]
          ↓
[Интеграция мессенджера]

    Очередь входящих запросов

    Очередь входящих запросов хранит данные, полученные от внешнего мессенджера.

    Когда пользователь отправляет сообщение, команду или выполняет другое действие, конкретная интеграция получает событие из API мессенджера и записывает его в очередь входящих запросов.

    На этом этапе РИМ ещё не обязан иметь полностью разобранное внутреннее событие. В очередь попадают данные входящего запроса, а их разбор и наполнение контекста выполняются позже, во время обработки очереди.

    В записи очереди фиксируются бот, данные запроса, дата записи, состояние обработки и информация об ошибке, если обработка завершилась неуспешно.

    Очередь сообщений к отправке

    Очередь сообщений к отправке хранит исходящие сообщения перед отправкой во внешний мессенджер.

    Сообщение может быть создано прикладной логикой бота или системной логикой РИМ: например, при аутентификации, отказе в доступе, приветствии пользователя или обработке команды.

    Сообщение не отправляется напрямую из обработчика бота. Сначала оно записывается в очередь отправки, затем отдельный обработчик выбирает его, подготавливает данные адресата, передаёт интеграции мессенджера и фиксирует результат отправки.

    Для исходящих сообщений РИМ может учитывать количество попыток отправки, дату успешной отправки, описание ошибки и режим хранения записи очереди.

    📥 Входящие запросы

    Входящий запрос — это внутренняя запись РИМ, созданная на основе события, полученного от мессенджера.

    Например, пользователь отправляет сообщение или команду боту. Интеграция мессенджера получает событие из API мессенджера, приводит данные к внутреннему формату РИМ и записывает их в очередь входящих запросов.

    Дальше входящий запрос обрабатывается внутренними механизмами РИМ:

    1. РИМ выбирает входящий запрос из очереди.
    2. Для запроса создаются контекст выполнения и контекст запроса.
    3. РИМ определяет бота, адресата, чат и инициатора сообщения.
    4. РИМ получает или создаёт сессию пользователя.
    5. Выполняется аутентификация и проверка доступа.
    6. Если запрос может быть обработан, он передаётся в прикладную логику бота.
    7. После завершения обработки РИМ фиксирует результат входящего запроса.
    [Событие мессенджера]
          ↓
[РИМ: интеграция мессенджера]
          ↓
[РИМ: очередь входящих запросов]
          ↓
[РИМ: менеджер входящих запросов]
          ↓
[РИМ: контексты, сессия и доступ]
          ↓
[Бот / прикладная логика]
          ↓
[РИМ: постобработка входящего запроса]

    Если на одном из системных этапов запрос не должен передаваться боту, РИМ завершает обработку без вызова прикладной логики. Например, это может произойти при отказе в доступе, продолжении сценария аутентификации или служебной обработке запроса.

    Контексты обработки

    Во время обработки входящего запроса РИМ использует два основных контекста: контекст выполнения и контекст запроса.

    Для каждого входящего запроса создаётся собственная пара контекстов. Эти контексты изолированы от обработки других запросов и используются как рабочее состояние текущей обработки.

    Контексты создаются после выбора входящего запроса из очереди и сопровождают его обработку до передачи в прикладную логику бота. По мере прохождения системных этапов РИМ наполняет и уточняет данные контекстов: определяет вид мессенджера, бота, адресата, чат, инициатора сообщения, сессию пользователя и результат проверки доступа.

    Контекст выполнения описывает окружение обработки. В нём хранятся данные, которые относятся не столько к самому сообщению, сколько к условиям его обработки: вид мессенджера, данные бота и данные адресата, которому могут отправляться ответные сообщения.

    Контекст запроса описывает сам входящий запрос. В нём хранятся исходные данные запроса, текст сообщения или команды, идентификатор чата, идентификатор инициатора, данные сессии и результат проверки доступа.

    Сессия, аутентификация и доступ

    После подготовки контекстов РИМ определяет сессию пользователя в рамках бота, чата и инициатора сообщения.

    Сессия позволяет хранить состояние взаимодействия пользователя с ботом между входящими запросами.

    После определения сессии РИМ выполняет аутентификацию и проверку доступа в соответствии с настройками бота и интеграции мессенджера.

    В зависимости от результата проверки РИМ может разрешить выполнение обработчика бота, продолжить диалог аутентификации, отправить сообщение об отказе в доступе или завершить обработку запроса без вызова прикладной логики.

    📤 Исходящие сообщения

    Исходящее сообщение — это сообщение, которое РИМ должен отправить пользователю через интеграцию мессенджера.

    Такое сообщение может быть сформировано в разных ситуациях: как ответ на входящий запрос пользователя, как часть сценария бота, как сообщение об отказе в доступе, как продолжение аутентификации или как результат работы системной логики РИМ.

    Сообщение не отправляется в мессенджер напрямую. Сначала оно записывается в очередь сообщений к отправке. Затем отдельный обработчик выбирает сообщение из очереди, подготавливает данные адресата, передаёт сообщение интеграции мессенджера и фиксирует результат отправки.

    Общий процесс выглядит так:

    1. РИМ или бот формирует исходящее сообщение для пользователя.
    2. Сообщение записывается в очередь сообщений к отправке.
    3. Обработчик очереди выбирает сообщение для отправки.
    4. РИМ подготавливает данные бота, адресата и параметры отправки.
    5. Интеграция мессенджера преобразует сообщение в формат API мессенджера.
    6. Интеграция отправляет сообщение через API мессенджера.
    7. РИМ фиксирует результат отправки: успех, ошибку, количество попыток и дату отправки.

    🚧 Границы ответственности

    Важно разделять ответственность интеграции мессенджера, прикладной логики бота и внутренних механизмов РИМ.

    Интеграция мессенджера отвечает за:

    • работу с API мессенджера;
    • получение входящих данных из мессенджера;
    • приведение данных мессенджера к внутренней модели РИМ;
    • дополнение контекста запроса данными конкретного мессенджера;
    • преобразование исходящих сообщений РИМ в формат API мессенджера;
    • отправку сообщений через API мессенджера;
    • работу с настройками подключения;
    • учёт технических особенностей конкретного мессенджера.

    Граница ответственности

    Интеграция мессенджера не предназначена для размещения бизнес-логики конкретного бота. Её роль — быть транспортным и адаптационным слоем между мессенджером и внутренними механизмами РИМ.

    Бот отвечает за прикладную логику:

    • обработку команд;
    • выполнение сценариев;
    • формирование ответов;
    • работу с бизнес-данными;
    • принятие решений на основе входящего запроса и пользователя, прошедшего проверку доступа.

    РИМ отвечает за внутреннюю обработку между интеграцией мессенджера и прикладной логикой бота:

    • запись входящих запросов в очередь;
    • запуск и выполнение обработки очереди входящих запросов;
    • создание контекста выполнения;
    • создание и обработку контекста входящего запроса;
    • выполнение системных обработчиков;
    • управление сессиями пользователей;
    • аутентификацию и проверку доступа;
    • передачу запроса в прикладную логику бота;
    • постобработку входящих запросов;
    • запись исходящих сообщений в очередь отправки;
    • обработку очереди сообщений к отправке;
    • фиксацию результатов отправки;
    • хранение или удаление записей очередей в зависимости от настроек.
